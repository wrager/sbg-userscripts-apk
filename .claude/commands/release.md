Выпустить релиз версии $ARGUMENTS (например: `/release 1.1.0`).

## Процесс

1. Определить предыдущий релиз — последний тег (`git describe --tags --abbrev=0`). Если тегов нет — все коммиты.
2. Проанализировать коммиты с момента предыдущего релиза (`git log <prev-tag>..HEAD --oneline`).
3. Сформировать `RELEASE_NOTES.md`:
   - Заголовок `# v{версия}`
   - Изменения, важные пользователям (фичи, заметные улучшения) — кратко, без технических деталей реализации
   - Фиксы предыдущих версий — одной строкой, без объяснения причин
   - Внутренние/dev изменения (рефакторинг, инфраструктура, сборка) — одной строкой без пояснений
   - Группировать по категориям если применимо
4. Обновить `versionMajor`/`versionMinor`/`versionPatch` в `app/build.gradle.kts` на новую версию. `versionCode` и `versionName` вычисляются автоматически.
5. Запустить полную проверку: `./gradlew ktlintCheck detekt testDebugUnitTest assembleDebug`
6. Закоммитить все изменения (RELEASE_NOTES.md, build.gradle.kts).
7. Создать тег `v<версия>`.

**Не пушить.** Пользователь пушит сам. Workflow `.github/workflows/release.yml` автоматически создаст GitHub Release с APK в ассетах.

**Релизный коммит** — только релизные файлы. Правки кода — отдельные коммиты ДО релиза.
